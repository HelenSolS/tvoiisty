# Сервер: архитектура, Docker, обработка примерок и поток картинок

Документ объединяет: как планировали строить сервер (Docker, сервисы), как обрабатываются примерки, где хранятся фото пользователя и образы, как они соединяются и как отправляются в Fal/KIE.

---

## 1. Планируемая инфраструктура (по документации)

По **docs/06-Infra-DevOps.md** для MVP предполагался один VPS с Docker Compose и следующими сервисами:

| Сервис | Назначение |
|--------|------------|
| **api** | Backend API (Express): приём запросов, роуты примерки/загрузки/настроек, вызов провайдеров. |
| **worker** | Отдельный воркер для тяжёлых задач (в текущей реализации примерка выполняется внутри api, отдельный worker не поднят). |
| **postgres** | БД: пользователи, media_assets, looks, tryon_sessions, настройки, логи. |
| **redis** | Очередь задач (в текущей реализации очередь анализа фото — in-memory в процессе api; redis в compose есть, но для очереди примерки не используется). |
| **minio** | S3-совместимое хранилище (в текущей реализации используется Supabase Storage и/или Vercel Blob, не MinIO на том же сервере). |
| **nginx** | Reverse proxy, SSL, раздача статики; таймауты для долгих запросов (upload, опрос примерки). |

Итог: в репозитории в **docker-compose.yml** реально описаны только **postgres**, **redis** и **backend** (один контейнер API). Nginx, MinIO и отдельный worker на практике либо вынесены (nginx на хосте), либо заменены внешними сервисами (Supabase/Blob вместо MinIO), либо не развёрнуты (worker). Ниже — как сделано сейчас и как планировалось.

## 2. Текущий Docker (что в репо)

**Файлы:** `Dockerfile`, `docker-compose.yml`.

- **postgres** — образ postgres:16, база tvoiisty_db, порт не пробрасывается наружу (доступ только из backend). Volume для данных.
- **redis** — образ redis:7, порт 6379, только внутри сети compose.
- **backend** — сборка из Dockerfile (Node 20, `npx tsx server.ts`), зависит от postgres и redis, env из `.env`, переменные БД переопределены для контейнерной сети (PGHOST=postgres и т.д.). Порт 3000 пробрасывается на хост.

На сервере деплой по **scripts/deploy-on-server.sh**: git pull, `docker compose up -d --build`. Перед backend снаружи обычно стоит **nginx** (на хосте): проксирование на localhost:3000, SSL для api.tvoiistyle.top, увеличенные таймауты для upload и долгих ответов (см. README, nginx-api-upload.conf.example).

## 3. Обработка примерок на сервере

3.1. Запрос приходит на **nginx** → проксируется на контейнер **backend** (Express).

3.2. **POST /api/tryon** — тонкий контроллер: валидация тела (person_asset_id, look_id или clothing_image_url), создание сессии в БД (pending), ответ 201 с tryon_id. Дальше обработка идёт **асинхронно в том же процессе** (async IIFE): не отдельный worker, не очередь в Redis — тот же Node-процесс backend.

3.3. В фоне backend: помечает сессию processing, по person_asset_id и look_id/clothing_image_url получает **два URL** (см. раздел 4), вызывает TryOnEngine (Fal → при сетевой ошибке retry Fal → fallback KIE), по успеху — mirror результата в хранилище и запись в media_assets + tryon_sessions или при сбое mirror — только сохранение URL в result_meta. Обновляет счётчики и токены.

3.4. Клиент результат получает через **GET /api/tryon/:id** (polling). Таймаут Fal на бэкенде 70 с; на фронте рекомендуемый лимит опроса не менее 90 с.

Итого: один контейнер backend обрабатывает и HTTP, и фоновую примерку; отдельный worker и Redis для примерки не используются. Очередь только для анализа загруженных фото (in-memory в backend).

## 4. Где хранятся фото и образы, как соединяются и как уходят в Fal/KIE

### 4.1. Разделение: «у пользователя» и «на сервере»

- **Локально у пользователя (браузер):** для удобства может храниться галерея выбранных фото (например, person_gallery в localStorage), ограниченный объём (например, последние 10–20), в сжатом виде. Это только для UX и быстрого выбора «своего» фото; для самой примерки этого недостаточно.
- **Для примерки обязательно:** фото человека должно быть **загружено на бэкенд** (POST /api/media/upload, type=person). Бэкенд сохраняет файл в **внешнее хранилище** (Supabase или Vercel Blob) и записывает запись в **media_assets** (id, type=person, original_url, storage_key, hash). Дальше в примерку передаётся **person_asset_id** — это ссылка на уже загруженное фото на сервере.
- **Образы одежды:** хранятся в том же хранилище (Supabase/Blob): при публикации товара или коллекции фронт загружает картинки через тот же POST /api/media/upload (type=clothing), в каталоге образов (looks) хранится main_asset_id → media_assets. То есть образы «подходящего для примерок размера» лежат в **облачном хранилище**, а не только на устройстве пользователя; размер/качество задаётся при загрузке (на клиенте допускается предварительное сжатие, например до 960 px по длинной стороне и JPEG 0.8).

### 4.2. Как «соединяются» свежезагруженное фото пользователя и образ

- Пользователь выбирает своё фото для примерки: это должен быть уже загруженный ассет (person_asset_id), т.е. один из тех, что ранее отправили через /api/media/upload и получили assetId.
- Пользователь выбирает одежду: либо **look_id** (образ из каталога — у него в БД есть main_asset_id → media_assets), либо **clothing_image_url** (прямой URL, например после uploadClothingImage для карточки магазина).
- При создании примерки (POST /api/tryon) бэкенд:
  - по **person_asset_id** достаёт из media_assets поле **original_url** (публичный URL фото человека в нашем хранилище);
  - по **look_id** — делает JOIN looks + media_assets и достаёт **original_url** образа одежды; по **clothing_image_url** — использует переданный URL как есть (если начинается с http).
- В итоге у backend есть два **публичных URL**: personUrl и clothingUrl. Никакой «склейки» файлов на сервере не делается: Fal и KIE принимают **два URL**, по ним сами скачивают картинки.

### 4.3. Как отправляем в Fal и KIE

- **Fal** (backend/falClient.ts): в тело запроса передаются только `person_image_url` и `clothing_image_url` (строки URL) и `preserve_pose: true`. Fal по этим URL сам забирает изображения.
- **KIE** (backend/kieClient.ts): в createTask передаётся input с `input_urls: [personUrl, clothingUrl]` (в основном потоке примерки с Express мы уже работаем с URL; в эндпоинте POST /api/generate-image KIE принимает base64, но это другой сценарий). KIE по URL или base64 сам получает картинки.

Требование: **оба URL должны быть доступны по HTTP/HTTPS с интернета** (Fal и KIE — внешние сервисы). Поэтому хранилище — обязательно с публичными URL (Supabase public bucket или Vercel Blob с access: 'public'). Локальные файлы только на диске сервера в Fal/KIE не отправить — нужны именно URL.

## 5. Схема потока картинок (кратко)

1. **Фото пользователя:** выбор в интерфейсе → (если ещё не загружено) загрузка через POST /api/media/upload → сохранение в Supabase/Blob → запись в media_assets → возврат assetId. В интерфейсе можно хранить локально до 10–20 превью для выбора, но для примерки всегда используется person_asset_id, т.е. фото уже на сервере.
2. **Образ одежды:** либо из каталога (look_id → main_asset_id → media_assets.original_url), либо URL после загрузки образа (clothing_image_url). В обоих случаях итог — URL на файл в нашем хранилище (или переданный https URL).
3. **Примерка:** POST /api/tryon с person_asset_id и look_id или clothing_image_url → backend по БД/телу получает personUrl и clothingUrl → отправляет оба URL в Fal (или KIE при fallback) → провайдер возвращает URL результата → backend зеркалирует результат в наше хранилище и пишет в tryon_sessions (или только result_meta.image_url при сбое mirror).
4. **Клиент:** опрашивает GET /api/tryon/:id и получает image_url (из media_assets или из result_meta).

## 6. Что доработать для полного соответствия плану (06)

- **Nginx:** описан на уровне «на сервере перед backend» и примеров конфига; в compose не входит — либо отдельный контейнер nginx, либо nginx на хосте (как сейчас по deploy-on-server.sh).
- **Worker:** если выносить долгую примерку в отдельный воркер, понадобится очередь (например Redis) и контейнер worker, который забирает задачи и вызывает тот же TryOnEngine; api тогда только создаёт сессию и ставит задачу в очередь.
- **Хранилище:** при переходе на MinIO/S3 на том же VPS нужно заменить вызовы Supabase/Blob в backend/storage.ts на клиент S3 и задать публичный доступ к бакету или раздачу через nginx/proxy, чтобы Fal/KIE могли скачивать по URL.
- **Очередь анализа фото:** при желании сделать её устойчивой к перезапускам — постановка задач в Redis и воркер, читающий из Redis и вызывающий aiPhotoPipeline.

Текущая реализация сознательно упрощена: один backend, внешнее хранилище (Supabase/Blob), примерка в процессе api, без отдельного worker и без Redis для очередей. Документ фиксирует и этот вариант, и целевой по 06-Infra-DevOps.
