# Параметры и настройки: что где хранится и кто чем управляет

Описание того, чем управляем в админке, что глобально на сервере, что локально для тестов и что может настраивать пользователь. Сейчас многое хранится локально; ниже — как сделано и как планировали сделать правильно.

---

## 1. Три уровня настроек

| Уровень | Где хранится | Кто меняет | Для чего |
|--------|--------------|------------|----------|
| **Глобальные (платформа)** | БД `app_settings` | Администратор через API | Модели по умолчанию, лимиты токенов, провайдеры — один набор на всё приложение. |
| **Локальные / админка (тесты и демо)** | `localStorage` в браузере (`tvoisty_admin_settings`) | Кто зашёл в админку (пароль 888) | Выбор моделей в выпадающем списке, промпты, fallback, «показывать выбор на главном» — только на этом устройстве. |
| **Пользовательские** | БД `users.preferences` (JSONB) | Авторизованный пользователь через «Настройки» | Тема, предпочитаемая модель, сцена и т.п. — у каждого пользователя свои. |

---

## 2. Глобальные настройки (сервер, БД)

**Таблица:** `app_settings` (ключ — строка, значение — JSONB). Инициализация: `backend/settings.ts` (INITIAL_SETTINGS), при первом запуске заполняется из `ensureAppSettings()`.

**API:** требуется авторизация и роль `admin`.

- **GET /api/admin/settings/global** — отдать все ключи и значения.
- **PUT /api/admin/settings/global/:key** — записать значение (тело: `{ "value": ... }`).

**Ключи и типы:**

| Ключ | Тип | Назначение |
|------|-----|------------|
| INITIAL_TOKENS | number | Стартовый баланс токенов у нового пользователя. |
| FREE_DAILY_IMAGES | number | Бесплатных примерок в день. |
| FREE_DAILY_VIDEOS | number | Бесплатных видео в день. |
| TOKENS_PER_IMAGE | number | Списание токенов за одну примерку. |
| TOKENS_PER_VIDEO | number | Списание токенов за одно видео. |
| TRYON_RESULT_TTL_DAYS | number | Срок хранения результата примерки (дни). |
| DEFAULT_IMAGE_MODEL | string | Модель примерки по умолчанию (например `fal-ai/nano-banana-pro/edit`). Используется бэкендом, если в запросе не передан `model_name`. |
| DEFAULT_VIDEO_MODEL | string | Модель видео по умолчанию (например `grok-imagine/image-to-video`). |
| ENABLED_IMAGE_PROVIDER | string | Провайдер примерки по умолчанию: `fal` или `kie`. |
| ENABLED_VIDEO_PROVIDER | string | Провайдер видео (например `fal`). |

**Где используются на бэкенде:**

- `backend/routes/tryon.ts` — при отсутствии `model_name` в теле берётся `getSetting('DEFAULT_IMAGE_MODEL')`.
- `backend/routes/generateImage.ts` — модель из тела или `getSetting('DEFAULT_IMAGE_MODEL')`.
- `backend/routes/generateVideo.ts` — модель из тела или `getSetting('DEFAULT_VIDEO_MODEL')`.
- `backend/services/providerRouter.ts` — `getSetting('ENABLED_IMAGE_PROVIDER')` для выбора провайдера (в сценариях, где используется роутер).

**Важно:** текущая админ-панель (AdminPanel) **не пишет** в эти ключи: кнопка «Сохранить» сохраняет настройки только в **localStorage**. Чтобы глобальные настройки реально менялись на сервере, админ должен вызывать PUT /api/admin/settings/global/:key (вручную, скриптом или из будущей версии админки).

## 3. Локальные настройки админки (localStorage)

**Ключ в браузере:** `tvoisty_admin_settings`. Чтение/запись: `services/adminSettings.ts` (getAdminSettings, setAdminSettings). Вход в админку: пароль 888 (заглушка).

**Назначение:** настройки «только для этого устройства» — тесты, демо, Лаборатория. Не синхронизируются с сервером и не действуют для других пользователей.

**Поля (тип AdminSettings в types.ts):**

| Поле | Описание |
|------|----------|
| provider | Провайдер по умолчанию для UI: `kie` \| `fal`. |
| defaultImageModel | Модель примерки по умолчанию для выбора на фронте. |
| defaultVideoModel | Модель видео по умолчанию для выбора на фронте. |
| imageModelChoice | Режим выбора модели примерки: `default_only` (без выпадающего списка) или `dropdown`. |
| imageModelsInDropdown | Список моделей в выпадающем списке для примерки. |
| imageBackupModel | Запасная модель при включённом fallback (при ошибке первой модели). |
| videoModelChoice | Аналогично для видео. |
| videoModelsInDropdown | Список моделей для видео в выпадающем списке. |
| videoBackupModel | Запасная модель для видео. |
| imagePromptMode | Режим промпта примерки: `default` \| `openai` \| `custom`. |
| imagePromptDefaultText | Текст стандартного промпта для примерки (режим default). |
| imagePromptCustom | Свой промпт (режим custom). |
| videoPromptMode | Режим промпта для видео. |
| videoPromptDefaultText, videoPromptCustom | Аналогично для видео. |
| imageFallbackEnabled | При ошибке первой модели автоматически пробовать запасную (например KIE → Fal). |
| showModelChoiceOnHome | **Локально для теста:** показывать выбор моделей (фото/видео) на главном экране. `false` = выбор только в Дополнительных настройках / Лаборатории. |

**Как связано с примеркой:** при создании примерки фронт может отправить в теле запроса `model_name` (например из `selectedImageModel`). Тогда бэкенд использует переданную модель. Если `model_name` не передан, бэкенд подставляет глобальную настройку из БД (`DEFAULT_IMAGE_MODEL`). То есть «модель по умолчанию» на фронте берётся из localStorage (админка), но итоговая модель в API может быть либо из тела запроса (как раз из этих локальных настроек), либо из глобальной настройки сервера.

**Кнопка «Вернуть стабильность»:** сбрасывает локальные настройки в дефолт (Fal Banana для картинки, выбранная модель для видео из пула, без выбора на главном) и сохраняет в localStorage.

**Отладочные логи / «багхистори»:** в проекте newstyle есть экран AdminDebugPanel (логи последних действий/ошибок). При переносе их UI эту функцию разумно считать частью нашей админки и параметров: либо только клиентский буфер логов (как у них), либо подпитка с бэкенда (например последние N записей из логов или отдельная таблица/ключ в `app_settings`). Отдельный бэкенд-сервис для багтрекера не нужен — достаточно раздела в админ-панели.

## 4. Пользовательские настройки (сервер, на пользователя)

**API:** GET /api/user/settings, PUT /api/user/settings. Требуется авторизация (Bearer token). Данные хранятся в `users.preferences` (JSONB).

**Поля (UserPreferences в services/userSettings.ts):**

| Поле | Описание |
|------|----------|
| theme | Тема оформления: `turquoise` \| `lavender` \| `peach`. |
| showModelSelector | Показывать ли пользователю выбор модели (если платформа это разрешает). |
| preferredScene | Предпочитаемая сцена (опционально). |
| defaultImageModel | Предпочитаемая модель примерки пользователя (опционально). |

Сейчас фронт может читать/писать эти настройки через `fetchUserPreferences` / `updateUserPreferences`; использование в логике примерки (подстановка модели пользователя) зависит от реализации экрана настроек и передачи модели в запрос примерки.

---

## 5. Что планировали сделать правильно

- **Глобальные настройки** должны быть единственным источником правды для платформы: модели по умолчанию, лимиты, провайдеры — в БД, редактируются только админом через API. Админка в UI должна либо читать их через GET /api/admin/settings/global, либо при сохранении «глобальных» полей вызывать PUT /api/admin/settings/global/:key, чтобы не дублировать логику в localStorage.
- **Локальные настройки** оставить только для того, что действительно «только для этого устройства»: например «показывать выбор моделей на главном (для теста)», список моделей в выпадающем списке для Лаборатории, возможно режимы промптов для демо. Всё, что должно влиять на всех пользователей (модель по умолчанию, лимиты), — только в глобальных настройках на сервере.
- **Пользовательские настройки** — тема, предпочитаемая модель (если разрешено платформой), сцена — хранить на сервере в `users.preferences` и использовать при формировании запросов примерки/видео (например подставлять defaultImageModel пользователя, если он задан и не переопределён явным выбором).

Итог: параметры мы описываем в трёх уровнях (глобально, локально для тестов, пользователь); сейчас многое живёт в localStorage, а «правильно» — глобальное на сервере в `app_settings`, пользовательское в `users.preferences`, локально только то, что реально только для устройства/теста.
