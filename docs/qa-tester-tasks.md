# Задания для тестировщика: TryOn API, Health, скрипт, fallback, pipeline

Цель: на основе реализованного кода оформить задания по проверке (чек-листы / тест-кейсы) по пяти пунктам. **Сами тест-кейсы формулирует тестировщик;** этот документ задаёт рамки и обязательные элементы заданий.

**Важно:** автотесты по заданию **обязательны только для ISSUE 4 и ISSUE 5**. Для пунктов 1–3 достаточно документации, endpoint’а и скрипта; автотесты там опциональны (при желании можно добавить проверки контракта, health, лога).

---

## Окружение

Перед составлением шагов уточнить:

- **Ручные проверки (ISSUE 1, 2, 3):** на каком URL поднимается сервер (localhost:4000 или тестовый хост, например `https://api.tvoiistyle.top`), нужны ли тестовые ключи/файлы (person/clothing для tryon).
- **Автотесты (ISSUE 4, 5):** как в проекте принято запускать тесты (`npm run test`, `npx vitest run ...`), что нужно для интеграционных тестов (PostgreSQL, переменные окружения, см. README и `tests/`).

**Не делать:** не менять код автотестов и скрипта без согласования с разработчиком; не писать новый код — только формулировки заданий и шаги проверки.

## ISSUE 1 — Контракт TryOn API

**Что проверяем:** соответствие реальных ответов API контракту из `docs/api-contract-tryon.md` (POST /api/tryon, GET /api/tryon/:id, upload).

**Правило:** контракт не меняется без версионирования (например `/api/v2/...`).

В задании должно быть явно:

| Элемент | Содержание |
|--------|-------------|
| Предусловия | Сервер поднят; при необходимости — тестовые `person_asset_id`, `look_id` или загрузка через upload. |
| Шаги | Вызовы POST /api/tryon, GET /api/tryon/:id, upload в соответствии с контрактом; проверка структур ответа (поля, типы, примеры из api-contract-tryon.md). |
| Ожидаемый результат | Ответы совпадают с описанными в контракте (поля `tryon_id`, `status`, `image_url`, `error` и т.д., типы значений). |
| Где смотреть результат | Postman / curl / браузер (сеть). |

Автотесты по контракту не обязательны; при желании можно добавить тесты, что реальные ответы API совпадают с описанным контрактом.

## ISSUE 2 — Health endpoint

**Что проверяем:** вызов GET /health, HTTP 200 и тело JSON: `status: "ok"` и при наличии — `service: "tryon-server"`.

В задании должно быть явно:

| Элемент | Содержание |
|--------|-------------|
| Предусловия | Сервер поднят (локально или тестовый хост). |
| Шаги | GET /health. |
| Ожидаемый результат | HTTP 200, JSON с полем `status: "ok"` и при наличии `service: "tryon-server"`. |
| Где смотреть результат | Postman / curl / браузер. |

Отдельный новый автотест не обязателен: проверка GET /health уже есть в smoke-тестах (`tests/smoke/server-viability.test.ts`).

## ISSUE 3 — Скрипт health и лог

**Что проверяем:** запуск скрипта, появление/обновление `logs/tryon-health.log` и формат записей (OK/ERROR, длительность, stage при ошибке).

В задании должно быть явно:

| Элемент | Содержание |
|--------|-------------|
| Предусловия | Сервер поднят; при необходимости — переменные окружения (URL, ключи). |
| Шаги | Запуск скрипта (команда из README или `npx tsx scripts/tryon-health-test.ts`). |
| Ожидаемый результат | В `logs/tryon-health.log` появляются/обновляются записи; формат содержит OK/ERROR, при ошибке — длительность, stage (если есть). |
| Где смотреть результат | Файл `logs/tryon-health.log`. |

Автотест не обязателен; при желании можно добавить тест вида «запустить скрипт и проверить, что в логе есть OK».
