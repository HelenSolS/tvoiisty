# 07. QA и автотесты

## 7.1. Роль тестировщика в процессе
На каждую фичу должен появляться:
- набор тест-кейсов
- автотесты на критические пути

## 7.2. Пирамида тестов (MVP)
- Unit (логика фильтров, валидации, статусы)
- Integration (API + DB)
- E2E (основные user flows)

## 7.3. Критические E2E сценарии
1) Пользователь без авторизации выбирает фото/образ → жмёт “Примерить” → авторизация → задача создаётся → статус обновляется → результат отображается.
2) Каталог образов: фильтры работают, подгрузка работает.
3) Выбор локации сохраняется и подставляется в примерку.
4) Кнопка “В магазин” логирует событие и открывает ссылку.
5) Загрузка образа магазином: проходит модерацию/отклоняется.

## 7.4. Definition of Done (DoD)
Фича считается готовой только если:
- код прошёл линтер и типы
- есть unit tests на ключевую логику
- есть интеграционный тест (если есть API)
- обновлены документы/README при необходимости
- PR прошёл review

## 7.5. Автотесты в CI
GitHub Actions обязаны запускать:
- lint
- typecheck
- unit tests
- integration tests
- build

Блокировать merge в main, если тесты не прошли.
