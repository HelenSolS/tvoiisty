# Функциональные требования (tvoiisty)

Единый документ функциональных требований к текущему интерфейсу tvoiisty. Основа — реализованное поведение (Express-бэкенд как основной сценарий).

---

## 1. Область действия

1.1. Документ описывает требования к веб-приложению виртуальной примерки (try-on) и генерации видео по результату примерки в репозитории tvoiisty.

1.2. В scope входят: загрузка и хранение медиа, создание и опрос сессий примерки, выдача результата пользователю, генерация видео по картинке, анализ загруженных фото, настройки платформы и пользователя.

1.3. Основной сценарий работы — фронт обращается к Express-бэкенду (POST/GET /api/tryon, POST /api/media/upload и др.). Альтернативный сценарий (часть запросов на Vercel serverless API) может предъявлять подмножество этих требований в зависимости от используемых эндпоинтов.

---

## 2. Пользователи и роли

2.1. Система должна различать роли: анонимный пользователь, авторизованный пользователь (client), администратор (admin).

2.2. Для демо допускается работа без авторизации для примерки и загрузки; списки «мои примерки», лайки образов, настройки пользователя и админ-настройки должны требовать авторизации и соответствующих прав.

2.3. Администратор должен иметь возможность просматривать и изменять глобальные настройки (модели по умолчанию, лимиты токенов и т.д.).

---

## 3. Медиа: типы и появление

3.1. Система должна поддерживать следующие типы медиа в реестре и хранилище: **person** (фото человека), **clothing** (фото одежды), **location** (локация/сцена), **tryon_result_image** (результат примерки), **tryon_result_video** (тип зарезервирован под видео).

3.2. Фото человека (person) должно появляться только через загрузку пользователем: выбор файла на фронте → отправка на бэкенд → сохранение в хранилище и регистрация в реестре медиа. Должна обеспечиваться дедупликация по хешу и типу: повторная загрузка того же файла не должна создавать дубликат в хранилище и в реестре.

3.3. Фото одежды (clothing) должно появляться: (a) через загрузку (аналогично person) при публикации товара/коллекции или загрузке образа; (b) через привязку к образу из каталога (look), где образ уже зарегистрирован в реестре по main_asset_id. Для примерки допускается передача URL одежды (clothing_image_url) без обязательной регистрации нового ассета в рамках данного запроса.

3.4. Результат примерки (tryon_result_image) должен появляться после успешной генерации провайдером (Fal или KIE): система должна пытаться сохранить картинку в своё хранилище (mirror по URL ответа провайдера) и зарегистрировать в реестре. Если сохранение в хранилище недоступно, система должна всё равно обеспечить выдачу картинки пользователю (например, через сохранение URL провайдера в метаданных сессии), с учётом возможного ограниченного срока жизни такого URL.

3.5. Видео по результату примерки в текущем сценарии может возвращаться пользователю только как URL в ответе на запрос генерации; требование сохранять видео в реестре медиа и в хранилище в рамках основного потока не предъявляется (опционально на будущее).

---

## 4. Медиа: хранение и доступ

4.1. Все загруженные и отзеркаленные файлы должны храниться во внешнем хранилище (Supabase Storage и/или Vercel Blob). Выбор конкретного провайдера хранилища определяется конфигурацией; при сбое основного хранилища допускается fallback на резервное, если оно настроено.

4.2. Реестр медиа (идентификатор, тип, URL, ключ в хранилище, хеш, MIME) должен храниться в БД. Доступ к URL для примерки и выдача результата пользователю должны осуществляться через этот реестр или через явно сохранённый URL в метаданных сессии (при сбое mirror).

4.3. Пользователь должен получать стабильный публичный URL для отображения загруженных фото и результата примерки. Основным источником URL результата примерки должно быть собственное хранилище; URL провайдера — только запасной вариант при недоступности хранилища.

---

## 5. Требования к файлам и изображениям

5.1. Система должна ограничивать размер загружаемого файла на приёме (например, не более 10 МБ). При превышении лимита должен возвращаться понятный ответ об ошибке.

5.2. На стороне клиента допускается предварительное сжатие/изменение размера изображений перед отправкой (например, ограничение по максимальной стороне и качеству JPEG) с целью сохранения достаточного качества для нейросетей и соблюдения лимитов размера запроса. Конкретные параметры (максимальная сторона, качество) задаются реализацией.

5.3. Требования к изображениям, передаваемым в нейросети (примерка, анимация), определяются контрактом внешних API (Fal, KIE). Система должна передавать в запросах к провайдерам параметры разрешения и соотношения сторон в соответствии с поддерживаемыми значениями API (например, aspect_ratio, resolution). Текущая реализация может использовать значения по умолчанию (например, 9:16 и 1K); изменение на иные значения (например, 2K) должно выполняться через конфигурацию или код в соответствии с документацией провайдера.

5.4. Текстовый промпт для примерки или видео не должен использоваться как единственный способ задания разрешения результата; разрешение должно задаваться параметрами API провайдера, где это предусмотрено.

---

## 6. Примерка (try-on)

6.1. Система должна предоставлять единый основной способ запуска примерки: создание сессии по идентификатору фото человека (person_asset_id) и либо идентификатору образа из каталога (look_id), либо URL картинки одежды (clothing_image_url). Обязательно наличие хотя бы одного источника одежды (look_id или clothing_image_url).

6.2. Создание сессии примерки должно возвращать клиенту идентификатор сессии (tryon_id) и начальный статус (например, pending). Генерация должна выполняться асинхронно; клиент должен получать результат и обновление статуса через опрос по идентификатору сессии (GET /api/tryon/:id).

6.3. Система должна использовать заданный порядок провайдеров примерки: основной провайдер (Fal) с резервным (KIE) при определённых типах ошибок. При сетевой ошибке основного провайдера допускается однократный повтор запроса перед переходом на резерв. При таймауте основного провайдера переход на резервный не обязателен (политика задаётся реализацией). Модель virtual-try-on не должна использоваться в качестве целевой модели примерки; при её указании должна выполняться подмена на согласованную модель (например, nano-banana).

6.4. Таймаут ожидания ответа от основного провайдера примерки должен быть не менее типичного времени ответа (например, 70 с). Таймаут опроса на стороне клиента должен быть больше таймаута провайдера (например, 90 с), чтобы пользователь не терял результат из-за преждевременного прекращения опроса.

6.5. При успешной генерации провайдером пользователь должен получить картинку независимо от того, удалось ли сохранить её в собственное хранилище: при сбое сохранения (mirror) система должна сохранить URL провайдера в метаданных сессии и отдать его в ответе GET /api/tryon/:id как image_url.

6.6. При ошибке примерки (включая таймаут и сбой резервного провайдера) пользователю должно показываться нейтральное сообщение об ошибке без раскрытия деталей провайдера, ключей и квот. Детальная информация должна записываться только в серверные логи для диагностики.

6.7. Должна обеспечиваться защита от повторных отправок запроса примерки во время выполнения одной примерки (например, блокировка кнопки или игнорирование повторных кликов до завершения текущей сессии). Идемпотентность по client_request_id при наличии авторизации должна возвращать существующую сессию без создания дубликата.

---

## 7. Генерация видео

7.1. Система должна позволять сгенерировать видео по URL изображения (как правило, результата примерки). Запрос должен содержать imageUrl; опционально — промпт и модель.

7.2. Генерация видео должна выполняться через внешний провайдер (KIE и др.); соотношение сторон видео должно соответствовать заданному формату (например, 9:16). Ответ должен содержать URL полученного видео (videoUrl). Сохранение видео в реестре медиа и в собственном хранилище может не входить в обязательные требования основного потока.

7.3. При ошибке или таймауте пользователю должно показываться понятное сообщение; детали провайдера не раскрываются.

---

## 8. Очередь анализа фото

8.1. После успешной загрузки фото (person, clothing, location) система должна ставить задачу на анализ изображения (модерация, описание, метаданные) в фоновую очередь. Ответ пользователю о результате загрузки должен отдаваться без ожидания завершения анализа.

8.2. Обработка очереди анализа может быть in-memory и не персистентной: при перезапуске процесса необработанные задачи могут теряться. Повторная загрузка того же файла (тот же хеш) не должна создавать дубликат в хранилище; при постановке новой задачи анализа для того же ассета и типа анализа уже существующий успешный результат может использоваться как кеш (повторный вызов внешнего сервиса не обязателен).

8.3. Примерка и генерация видео не должны обрабатываться через эту очередь; они выполняются в рамках запроса создания сессии или запроса генерации видео.

---

## 9. Каталог образов и лайки

9.1. Система должна предоставлять каталог образов (looks) с привязкой к медиа (main_asset_id). Должна быть возможность получать список образов с учётом сортировки (например, по дате, лайкам, количеству примерок) и фильтра «только понравившиеся» для авторизованного пользователя.

9.2. Авторизованный пользователь должен иметь возможность отмечать образы как понравившиеся (лайк) и снимать лайк. Количество лайков образа должно отображаться и храниться консистентно.

9.3. При успешной примерке по образу (look_id) счётчик примерок по этому образу должен увеличиваться.

---

## 10. Токены и учёт

10.1. Система может вести учёт токенов пользователя (списание за примерку, начисление). Списание должно фиксироваться после успешного завершения примерки (status completed). Конкретные правила (начальный баланс, стоимость примерки/видео) задаются настройками.

10.2. Список «мои примерки» для авторизованного пользователя должен возвращать сессии с привязкой к образам и URL результата без раскрытия внутренних идентификаторов провайдеров.

---

## 11. Настройки

11.1. Должны храниться глобальные настройки платформы (модели по умолчанию для картинки и видео, лимиты, флаги провайдеров и т.д.) и при необходимости настройки пользователя (предпочтения). Администратор должен иметь возможность их просматривать и изменять в рамках отведённых полей.

11.2. Проверка доступности хранилища (GET /api/media/upload/check) должна возвращать статус в зависимости от наличия конфигурации (например, Supabase или Vercel Blob); при отсутствии конфигурации — ответ с кодом, указывающим на недоступность загрузки.

---

## 12. Безопасность и ограничения

12.1. Ключи и секреты внешних сервисов (Fal, KIE, OpenRouter, хранилища) не должны передаваться на клиент и не должны попадать в сообщения об ошибках, отображаемые пользователю.

12.2. API примерки и выдачи результата должны быть стабильными: изменения контракта (тело запроса, формат ответа) без версионирования (например, /api/v2/...) не допускаются без отдельного решения.

12.3. CORS и доступ к API должны настраиваться под доверенные источники (фронт приложения); перечень разрешённых origin задаётся конфигурацией.
