# План восстановления: наш движок за основу, их вид — для инвестора

Цель: **наша примерочная (движок) + их интерфейс (внешний вид)**. Сначала переносим их фронт и проверяем, что вид устраивает инвестора; потом приводим в порядок сервер. Так и судим: по стабильности — наш движок, по виду — их UI.

---

## Главное по UX: быстрая примерка и вкладки

**Режим быстрой примерки (приоритет)**  
Человек заходит → залил своё фото → залил свой образ (одну картинку) → получил примерку. Без каталога и лишних шагов. Важно:
- **ловить результат примерки** (наш API уже отдаёт `image_url` по `GET /api/tryon/:id`) и **показывать** его в их UI;
- **сохранять в историю примерок** так же, как для обычного сценария: каждая примерка у нас уже пишется в `tryon_sessions`; фронт должен подтягивать историю с бэкенда (см. ниже).

**Полный режим (когда пользователь захочет)**  
Переход в «выбор образов» с вкладками, как у нас было:
- **Главная** — лучше назвать **«Мои фото»**: свои фото человека и свой «гардероб» (загруженные образы), откуда можно быстро слать две картинки в примерку.
- **История** — список завершённых примерок (данные с нашего бэкенда: `GET /api/history` или эквивалент).
- **Образы** — каталог образов (наш `GET /api/looks`).

**Багхистори (debug-логи)**  
У них в UI есть панель с логами (AdminDebugPanel, «!тест админ»). Отдельного бэкенда для багтрекера у них нет — по сути это просмотр отладочных сообщений. Имеет смысл считать это **нашими параметрами/админкой**: показ логов и ошибок в админ-панели, при необходимости подпитка с бэкенда (например, последние N записей из логов или из `app_settings`/отдельной таблицы). То есть их «багхистори» живёт внутри нашей системы настроек и админ-функций.

**Внешний вид «как у них» (референс для инвестора)**  
При переносе их фронта сохраняем именно такой визуальный стиль:
- **Большие картинки, удобно смотреть** — зоны для фото человека, одежды и результата примерки крупные, по центру; превью и результат занимают основную площадь экрана.
- **Не доминирующий фон** — светлый (белый, очень светло-серый, лёгкий лавандовый градиент), без тяжёлых цветов и перегруженного оформления; контент и картинки в приоритете.
- **Всё лайтово и воздушное** — много воздуха, скруглённые карточки, мягкие тени, минимум декора; интерфейс не давит, приятно пользоваться.

При любых доработках (наши компоненты, их компоненты) этот стиль не ломаем: большие картинки, светлый ненавязчивый фон, воздушная вёрстка.

**Темы (фоны)** — как и у нас: **персик, мята, лаванда**. У них добавлены свои акценты; берём их дизайн целиком. **Дизайн утверждён — мы берём его.**

---

## Принцип

- **Движок (бэкенд, пайплайн примерки)** — только наш (tvoiisty). Не судим по внешнему виду; стабильность и логика уже проверены.
- **Внешний вид** — берём от них (newstyle): экраны, шаги, оформление. **Утверждён:** их дизайн принимаем за основу.
- **Сервер** — настраиваем после того, как фронт привязан к нашему API и вид утверждён.

---

## Что нужно от тебя, чтобы начать (Фаза 1)

Чтобы приступить к переносу их фронта и привязке к нашему API, достаточно:

1. **Доступ к коду newstyle** — он уже есть (например `/Users/lena/newstyle`). Если работаем в репо tvoiisty, решаем: копируем их фронт в tvoiisty (отдельная папка/ветка) или правим у них и только потом переносим. Обычно удобнее скопировать их UI в наш репо и там менять вызовы на наш API.
2. **Наш бэкенд под рукой** — локально (`npm run server` в tvoiisty, с поднятыми postgres/redis по docker-compose или своими средствами) или тестовый URL, куда будет стучаться их фронт. Нужен рабочий `.env` (Fal, KIE, хранилище, БД).
3. **Команда «начинаем Фазу 1»** — явно сказать, что дизайн утверждён и можно переходить к техническому переносу: замена их API на наш, проверка быстрой примерки и истории в dev.

Ничего больше от тебя не требуется. Дальше: берём их компоненты, подключаем к нашему API, прогоняем сценарий в dev и итеративно поправляем.

## Фаза 1. Перенос их фронта и привязка к нашему API (сначала это)

**Цель:** один раз привязать их UI к нашему бэкенду (tvoiisty), проверить, что внешний вид такой, как надо инвестору. Движок при этом уже наш — запросы идут в наш API.

### 1.1. Где крутится бэкенд на этом этапе

- Вариант A: **локально** — у тебя `npm run server` в tvoiisty, фронт (их код) с `VITE_API_BASE_URL=http://localhost:3000` (или твой порт).
- Вариант B: **тестовый/существующий хост** — если наш API уже развёрнут (api.tvoiistyle.top или другой URL), фронт просто указывает на него.

На этом этапе сервер можно не трогать: работаем против уже работающего нашего API (локально или тестовый).

### 1.2. Что сделать с их фронтом

1. **Взять их фронт** (репозиторий/папка newstyle): их React-компоненты, экраны, стили, шаги (загрузка фото → выбор одежды → результат → видео).
2. **Заменить вызовы API на наш контракт:**
   - Загрузка фото человека → **POST /api/media/upload** (type=person или как у нас), из ответа брать **assetId** (это наш **person_asset_id**).
   - Список образов → **GET /api/looks** (формат ответа подстроить под их компоненты, если нужно).
   - Запуск примерки → **POST /api/tryon** с телом: **person_asset_id**, **look_id** и/или **clothing_image_url** (наш контракт, см. api-contract-tryon.md). Никаких photoId/user_photos.
   - Результат → **GET /api/tryon/:id** (polling до status completed), показ **image_url** из ответа. При сбое mirror у нас уже отдаётся image_url из result_meta.
   - **История примерок** — после каждой примерки сессия уже сохраняется у нас в `tryon_sessions`. Их фронт тянет историю через **GET /api/history** (формат: sessionId, imageUrl, videoUrl, createdAt, lookId, hasVideo). У нас есть **GET /api/my/tryons** (другой формат); либо добавить **GET /api/history** с такой же формой ответа из наших `tryon_sessions`, либо на фронте маппить ответ `/api/my/tryons` в их формат.
   - Видео (если есть в их UI) → **POST /api/generate-video** с **imageUrl** (URL результата примерки), в ответе **videoUrl**.
3. **Убрать зависимость от их бэкенда:** не вызывать их POST /api/tryon, их X-User-Id в том виде, в котором обязателен у них. У нас для демо user_id может быть null; при необходимости передаём заголовок или не передаём — по нашему API.
4. **Одежда «своя» (не из каталога):** если у них есть сценарий «загрузил фото вещи», то: загрузка через наш **POST /api/media/upload** (type=clothing), в примерку передаём полученный **url** в поле **clothing_image_url**. У нас это уже поддерживается.
5. **Собрать и запустить их фронт** (npm run build / npm run dev), в настройках указать базовый URL нашего API. Пройти сценарий: загрузка фото → выбор образа или своей одежды → примерка → результат. Убедиться, что запросы уходят в наш бэкенд и ответы обрабатываются.

Итог фазы 1: **один рабочий сценарий: их экраны + наш движок (наш API)**. Никакой «их» сервер не нужен.

### 1.3. Проверка «вид для инвестора»

- Пройти все экраны их UI: загрузка фото, выбор одежды, результат, опционально видео.
- Зафиксировать: это тот самый вид, который нужен инвестору (или список правок по виду).
- Если инвестор подтверждает — переходим к серверу. Если нужны правки по виду — делаем их на их фронте, не трогая наш движок.

## Фаза 2. Сервер под наш стек (после утверждения вида)

**Цель:** один раз развернуть наш бэкенд на целевом сервере, без их контейнеров и конфликтов портов. Фронт (уже привязанный к нашему API) отдаётся с того же хоста или с Vercel/отдельного домена.

### 2.1. План по серверу

Использовать **docs/SERVER-TAKEOVER-PLAN.md**:

1. Остановить их Docker Compose (освободить порты).
2. Поднять только наш compose: postgres, redis, backend (порт 3000 на localhost).
3. Nginx на хосте: 80/443 → proxy на backend, увеличенные таймауты для /api/.
4. В .env на сервере: ключи Fal, KIE, хранилище (Supabase или Blob), при необходимости OpenRouter.
5. Проверки: /health, /api/media/upload/check, одна примерка до конца.

### 2.2. Фронт на прод

- Либо **статику их фронта** раздаёт nginx с этого же сервера (сборка их приложения, `VITE_API_BASE_URL` = https://api.tvoiistyle.top или ваш домен).
- Либо фронт на **Vercel/другом хосте**, в настройках API URL = наш развёрнутый API. Тогда сервер занимается только API.

После этого «крутой движок» — наш, работает на нашем сервере; вид — их, привязан к нашему API.

|---|----------|-----------|
| 1 | Взять их фронт (newstyle), заменить вызовы на наш API (person_asset_id, look_id, clothing_image_url, GET /api/tryon/:id, при необходимости /api/generate-video). | **Сделано:** app-newstyle в репо, вызовы переведены на наш контракт. |

| 3 | Запустить наш бэкенд локально (или использовать уже развёрнутый тестовый), поднять их фронт с base URL на наш API. | Один сквозной сценарий: их вид + наш движок. |
| 4 | Пройти сценарий (быстрая примерка, затем вкладки: Мои фото, История, Образы), убедиться что примерка и при необходимости видео работают. | Стабильность на нашей стороне. |

## Что не делаем

- Не доводим **их** бэкенд до идеала — движок только наш.
- Не судим о качестве по одному внешнему виду — за основу берём нашу примерочную, вид дополняем их интерфейсом.
- Не трогаем сервер и их контейнеры **до** того, как фронт привязан к нашему API и вид проверен/утверждён.

Так мы восстанавливаем «крутой движок» (наш) и даём инвестору нужный вид (их фронт), с понятным порядком: сначала перенос фронта и проверка вида, потом сервер.
