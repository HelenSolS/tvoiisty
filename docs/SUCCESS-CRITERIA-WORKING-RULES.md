# Критерии успеха и рабочие правила

Чтобы не сбиться и держать курс: что считаем успехом, как работаем, что делаем перед любым шагом.

---

## Принципы (чтим всегда)

1. **Не ломаем то, что работает.** Рабочий функционал не трогаем без явного запроса. Любое изменение — только по задаче, с пониманием последствий.
2. **Идём строго по плану.** План восстановления — docs/PLAN-RECOVERY-ENGINE-AND-UI.md. Сначала фронт + наш API (фаза 1), потом сервер (фаза 2). Не перескакиваем, не меняем порядок без обсуждения.
3. **Проверяем результат.** После каждого шага — явная проверка: сценарий прошёл, ничего не сломалось. Без проверки шаг не считается завершённым.
4. **Работаем только в dev.** Всё тестируем в dev-окружении (локально или dev-ветка). В прод/main — только после проверки и явного решения.
5. **Сложности сразу обсуждаем.** Неприятности, риски, расхождение с планом — не «доделаю потом», а сразу ставим в известность и решаем: менять план, откатить, или принять осознанно.
6. **Каноны чтим.** Провайдеры/модели, контракт API, архитектура примерки — по документам. Не меняем без сверки с каноном и без явного решения.

## Действия: как «вспомнить» критерии перед работой

Перед началом задачи или сессии:

1. **Открыть план.** Если задача про перенос фронта или сервер — перечитать актуальный шаг в PLAN-RECOVERY-ENGINE-AND-UI.md (или SERVER-TAKEOVER-PLAN.md).
2. **Свериться с каноном.** Если трогаешь примерку, провайдеры, модели или API — открыть CANON-PROVIDERS-AND-MODELS.md и api-contract-tryon.md; не менять контракт и порядок провайдеров без решения.
3. **Проверить, что не ломаем.** Перед правкой: какой сценарий сейчас работает? После правки: тот же сценарий должен пройти (плюс новая фича, если есть).
4. **Окружение — только dev.** Запуск и тесты — локально или на dev. В main/прод не пушить и не деплоить без явного «можно в прод».
5. **Сложность или риск — сразу сказать.** Неясность по плану, конфликт с каноном, «а давайте заодно переделаем» — останавливаемся и обсуждаем.

После задачи:

6. **Кратко зафиксировать результат.** Что сделано, что проверили, что осталось (если есть). Чтобы в следующий раз не забыть контекст.

---

## Короткий чеклист на каждый шаг

- [ ] Задача из плана или явно согласована.
- [ ] Каноны (провайдеры, API) не нарушаю.
- [ ] Проверка в dev запланирована / сделана.
- [ ] Рабочий сценарий не сломан.
- [ ] Сложности и риски — озвучены.

Если все пункты — да, шаг можно считать выполненным. Если что-то «нет» — сначала закрыть вопрос (обсудить, свериться с документом, откатить), потом продолжать.
