# Проверки при каждом деплое (минимум, без избыточности)

Один и тот же набор проверок при каждом деплое — без лишних шагов.

---

## Бэкенд (tvoisty / api.tvoiistyle.top)

| Проверка | Когда | Как |
|----------|--------|-----|
| Unit + integration тесты | При push/PR в `dev` | Уже есть: GitHub Actions **Backend Tests** (`npm run test`, `npm run coverage`). Мержить только при зелёном статусе. |
| Доступность после деплоя | Сразу после деплоя | Health check по URL деплоя. На хостинге (Vercel/Railway/др.) указать **Health Check URL**: `https://api.tvoiistyle.top/health` (или свой домен). Успех = HTTP 200 и тело содержит `"status":"ok"`. |

Опционально: в GitHub Actions есть workflow **Deploy verify** (ручной запуск): проверяет GET /health по URL API после деплоя. Запускать при необходимости из вкладки Actions.

---

## Фронтенд (tvoiistyle)

| Проверка | Когда | Как |
|----------|--------|-----|
| Сборка | При каждом деплое | На Vercel (или другом хостинге) в настройках сборки указать команду: **`npm run ci`** (или `npm run build`). Деплой не создаётся, если сборка упала. |
| API доступен для фронта | По желанию после деплоя | Опционально: один раз после деплоя выполнить `ALLOW_API_CHECK=1 ./scripts/check-deploy.sh` в репо tvoiistyle (проверка GET /health бэкенда). Не обязательно в автоматическом режиме при каждом деплое. |

Дополнительно не гонять при каждом деплое: e2e, ручные сценарии — только по необходимости или по расписанию.

---

## Итог: что обязательно

- **Бэкенд:** CI тесты зелёные на `dev` перед мержем; после деплоя — успешный ответ на GET /health (через настройку хостинга).
- **Фронт:** при деплое выполняется `npm run ci` (сборка); успешный деплой = сборка прошла.

Этого достаточно для стабильного деплоя без избыточных проверок.
