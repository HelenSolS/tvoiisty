# DEPLOYMENT_BOUNDARIES

Границы деплоя и окружения. Менять их без явного согласования запрещено.

## Роли платформ

| Платформа | Назначение |
|-----------|------------|
| **Vercel** | Деплой frontend только. Подключён к репозиторию. |
| **Выделенный сервер** | Backend: API, очереди, воркеры, storage-оркестрация, AI pipeline. |

## Запреты

- **Backend не на Vercel/serverless.** Очереди, воркеры, работа со storage и AI должны оставаться на выделенном backend-сервере.
- **Не менять топологию деплоя** без явного указания (например, не «упрощать» деплой по своему усмотрению).
- **Не переименовывать, не удалять и не переиспользовать env-переменные** без явного согласования.

## Переменные окружения

- В репозиторий не попадают: API keys, пароли БД, токены сервисов, файлы `.env`.
- Допустимо добавлять только `.env.example`.
- Реальные секреты — только в окружении сервера/CI, не в коде и не в логах.

## Итог

- Frontend → Vercel.
- Backend (Node.js, Redis, очереди, воркеры, Supabase Storage, Fal/KIE) → выделенный сервер.
- Изменения в способе деплоя и наборе env — только по решению владельца проекта.
