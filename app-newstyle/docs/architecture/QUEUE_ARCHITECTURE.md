# QUEUE_ARCHITECTURE

## 1. Назначение очередей

Очереди используются для выполнения долгих и ресурсоёмких операций.

Причины:

- генерация изображений может занимать до 60 секунд
- внешние AI-сервисы могут отвечать медленно
- сервер не должен блокироваться.

Очереди позволяют:

- обрабатывать задачи асинхронно
- масштабировать систему
- реализовать retry и fallback.

## 2. Используемая технология

Очереди реализуются через:

- **Redis**
- **BullMQ**

Redis используется только как брокер задач.

## 3. Основные очереди

Система содержит четыре очереди.

### upload_queue

**назначение:**

обработка пользовательских фото после загрузки.

**операции:**

- сжатие изображения
- проверка размера
- загрузка в storage.

### tryon_queue

**назначение:**

генерация примерки.

**операции:**

- подготовка изображений
- вызов Fal
- retry
- fallback на KIE
- сохранение результата.

### video_queue

**назначение:**

генерация видео.

**операции:**

- получение изображения
- генерация видео
- сохранение результата.

### cleanup_queue

**назначение:**

удаление устаревших данных.

**операции:**

- удаление фото старше 15 дней
- очистка временных файлов.

## 4. Pipeline try-on

Процесс генерации выглядит так:

```
client
   ↓
POST /api/tryon
   ↓
создание tryon_session
   ↓
enqueue tryon job
   ↓
worker
   ↓
Fal try-on
   ↓
если ошибка
   ↓
retry
   ↓
fallback KIE
   ↓
сохранение результата
   ↓
обновление tryon_session
```

## 5. Retry правила

Fal может возвращать ошибки.

Правила:

- **retry:** 3 попытки
- **интервал:** 5 секунд.

## 6. Fallback

Если Fal не отвечает или возвращает ошибку:

используется KIE.

## 7. Ограничение параллелизма

Чтобы не перегружать сервер:

- **max concurrent jobs:** 5

## 8. Статусы задач

Задачи могут иметь статусы:

- pending
- processing
- completed
- failed
