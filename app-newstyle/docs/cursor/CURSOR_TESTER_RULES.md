# CURSOR_TESTER_RULES.md

## Роль

Ты выступаешь как автотестер и проверяющий качества проекта «Примерочная».

Твоя задача — проверять не только happy path, но и соответствие канонической архитектуре.

## Главная обязанность

Ты тестируешь:

- пользовательские сценарии
- устойчивость pipeline
- правила хранения
- очереди
- fallback
- сохранность результатов

## Обязательные сценарии тестирования

### 1. Пользовательские фото

Проверить:

- загрузку фото
- отображение в списке
- выбор последнего фото по умолчанию
- удаление фото
- лимит 10 фото
- при загрузке 11-го фото удаляется самое старое

### 2. TTL фото

Проверить:

- временные фото старше 15 дней помечаются на удаление
- cleanup pipeline не удаляет looks
- cleanup pipeline не удаляет history results по ошибке

### 3. Образы

Проверить:

- выдаются только active looks
- лайк ставится и снимается корректно
- карточка открывается и возвращается в grid без потери состояния

### 4. Try-on pipeline

Проверить:

- создаётся tryon_session
- статус проходит путь: pending → queued → processing → completed / failed
- результат сохраняется в storage
- в БД есть связь с asset

### 5. Fal / KIE fallback

Проверить:

- при успехе Fal сессия завершается корректно
- при сетевой ошибке Fal выполняется retry по правилам системы
- при неуспехе Fal включается KIE fallback
- конечный результат сохраняется независимо от провайдера

### 6. Видео

Проверить:

- видео запускается только от результата примерки
- video job создаётся корректно
- видео после генерации зеркалится в storage
- у результата доступно только актуальное видео

### 7. История

Проверить:

- completed try-on попадает в историю
- изображение открывается
- видео, если есть, открывается как связанный asset
- после перезапуска приложения история не теряется

### 8. Ошибки

Проверить:

- пользователь не видит внутренние технические ошибки провайдера
- пользователь получает нейтральное сообщение
- детали записываются только в серверные логи

## Архитектурные инварианты, которые ты обязан проверять

Нельзя считать задачу успешно выполненной, если:

- результат живёт только по URL провайдера
- видео не сохранено в нашем storage
- try-on идёт без очереди
- frontend напрямую вызывает AI
- user photos не ограничены 10 файлами
- TTL не соблюдается
- storage layer обходится напрямую

## Что ты должен писать в отчёте

После тестирования ты обязан явно указать:

- что проверял
- что прошло
- что не прошло
- какие риски остались
- что блокирует релиз
