# GIT_WORKFLOW_RULES.md

## Общий принцип

Репозиторий проекта является каноническим источником кода. Любые изменения в нём должны происходить строго через контролируемый workflow.

Цель этого правила:

- защитить стабильную версию проекта
- исключить случайные изменения
- обеспечить прозрачность разработки

## Структура веток

В проекте используются две основные ветки: **main** и **dev**.

### main

**Назначение:** стабильная версия проекта.

**Правила:**

- пушить может только владелец проекта
- Cursor не имеет права пушить в main
- любые изменения попадают в main только после ручного решения владельца

### dev

**Назначение:** ветка разработки.

**Правила:**

- Cursor может готовить изменения в dev
- Cursor может создавать pull request
- Cursor может пушить в dev только после разрешения

## Pull Request workflow

Правильный порядок работы:

1. Cursor анализирует задачу
2. Cursor предлагает план изменений
3. изменения согласуются
4. Cursor делает изменения в dev
5. создаётся Pull Request
6. владелец проекта проверяет изменения
7. только после этого изменения могут попасть в main

## Что Cursor запрещено

Cursor запрещено:

- пушить в main
- удалять ветки
- переписывать историю git
- делать force push
- менять структуру репозитория без согласования
- удалять legacy код

## Работа с приватным репозиторием

Репозиторий **https://github.com/HelenSolS/newstyle** является приватным.

Cursor взаимодействует с ним через:

- GitHub CLI
- или через встроенную интеграцию

## Работа с Vercel

Vercel подключён к репозиторию.

**Правила:**

- Vercel деплоит frontend
- backend не должен деплоиться через Vercel
- backend работает на сервере

## Правило коммитов

Коммиты должны быть:

- маленькими
- логически изолированными
- понятными

**Пример:**

```
feat: add tryon queue worker
fix: correct media asset storage path
refactor: isolate storage service
```

## Правило безопасности

В репозиторий запрещено добавлять:

- API keys
- database passwords
- service tokens
- .env файлы

Разрешено добавлять только:

- .env.example

## Важное правило

Cursor не должен автоматически удалять старый код. Любые изменения legacy частей должны обсуждаться отдельно.

## Дополнительное правило для Cursor

Перед любым изменением Cursor должен:

- описать изменения
- перечислить файлы
- дождаться подтверждения
