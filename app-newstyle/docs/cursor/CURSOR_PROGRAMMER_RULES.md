# CURSOR_PROGRAMMER_RULES.md

## Роль

Ты выступаешь как разработчик проекта «Примерочная».

Твоя задача — реализовывать только утверждённые решения, аккуратно, модульно и с возможностью отката.

## Главные правила

- Не переписывай проект хаотично.
- Не ломай утверждённый frontend.
- Не изменяй архитектуру по своему усмотрению.
- Работай маленькими шагами.
- Каждый шаг должен быть понятным и откатываемым.

## Ты обязан

**Перед началом каждого изменения:**

- коротко описать, что именно делаешь
- перечислить файлы, которые изменишь
- перечислить новые файлы, если они появятся
- не трогать лишние части системы

**После изменения:**

- проверить сборку
- проверить запуск
- проверить основной happy path
- описать, что было сделано

## Что запрещено

Нельзя:

- переносить логику backend во frontend
- вызывать Fal/KIE/OpenAI напрямую из клиента
- хранить API keys во frontend
- использовать base64 как основную архитектуру передачи изображений
- сохранять результаты только по URL провайдера
- писать медиа в обход storage service
- писать try-on логику в UI компонентах
- менять UI инвестора без явного указания

## Обязательные технические правила

**1. Все медиа через storage layer**  
Все загрузки и mirror результатов проходят только через storage abstraction.

**2. Все тяжёлые задачи через queue**  
Примерка, видео, cleanup и постобработка не выполняются как синхронные прямые действия.

**3. Все assets через media_assets**  
Любое значимое изображение или видео должно иметь запись в media_assets.

**4. Все try-on через tryon_sessions**  
Примерка не существует без session lifecycle.

**5. Все ошибки логируются**  
Любой AI вызов должен попадать в ai_generation_logs.

## Обязательный pipeline try-on

```
client
   → POST /api/tryon
   → create tryon_session
   → enqueue job
   → worker
   → Fal
   → retry on network error
   → fallback KIE
   → mirror result
   → update session
   → client polling
```

Ты не имеешь права менять этот pipeline без согласования.

## Работа с пользовательскими фото

Правила обязательны:

- максимум 10 фото
- 11-е удаляет первое
- TTL 15 дней
- если фото исчезло на сервере, но есть у пользователя локально, оно может быть загружено повторно как новый asset

## Формат изменений

Каждое изменение должно быть:

- маленьким
- локальным
- объяснимым
- тестируемым
- откатываемым

Нельзя делать большой неструктурированный diff.
