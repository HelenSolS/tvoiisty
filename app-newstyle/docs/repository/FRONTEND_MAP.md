# FRONTEND_MAP — Карта канонического frontend

Эти файлы и папки образуют утверждённый инвестором UI. **Не переписывать, не менять структуру экранов.**

---

## 1. Папки и файлы frontend (не менять структуру)

```
components/           # Все UI-компоненты основного потока и общие
  Step1UploadUser.tsx # Экран «Мои фото»
  Step2UploadClothing.tsx # Экран «Образы»
  Step3Result.tsx     # Экран «Результат примерки»
  Step4Video.tsx      # Экран «Видео»
  LookScroller.tsx    # Экран «История примерок»
  FullscreenPreview.tsx # Раскрытие карточки во весь экран (общее)
  Dropzone.tsx        # Загрузка файла (общее)
  Header.tsx          # Шапка, меню, навигация по шагам
  AuthModal.tsx       # Модалка авторизации
  AdminPanel.tsx      # Настройки (тема, пресеты и т.д.)
```

Корень:

- `App.tsx` — маршрутизация шагов (0–5), состояние, обработчики. Не менять последовательность шагов и переходы.
- `types.ts` — типы, enum’ы, `AuthState`, `LookHistoryItem`. Можно расширять под API, не ломая текущие экраны.
- `translations.ts` — тексты интерфейса (RU и др.). Только правки текста/переводов.
- `constants.tsx` — лимиты, пресеты, MOCK_SHOPS. При интеграции: убрать секреты (CUSTOM_SERVICE_*); константы UI и лимиты согласовать с backend.

| Экраны из MVP | Файл(ы) | Назначение |

## 3. Компоненты, которые нельзя менять по структуре

- **Step1UploadUser** — блок «МОИ ФОТО», галерея горизонтальная, кнопка «+», удаление, выбор фото, вызов `onUpload(img)`.
- **Step2UploadClothing** — блок галереи образов, переключатель grid/scroll, кнопка «+», превью и удаление своих загрузок, клик по образу → `onUpload(item.imageUrl)` (примерить).
- **LookScroller** — заголовок «История», переключатель grid/list, кнопка «+», карточки с лайком, удалением, скачиванием, видео; клик по карточке → fullscreen.
- **FullscreenPreview** — полноэкранное превью по клику, закрытие по кнопке. Не заменять отдельной страницей или модалкой с другим поведением.
- **Dropzone** — выбор файла → callback с base64. При интеграции: вызывать upload на backend и передавать в callback уже `url` (или `id`+`url`), не меняя визуал и поток «выбрал файл → появилось в галерее».

---

## 4. Что можно менять при интеграции (без смены структуры экранов)

- **Источник данных:** заменить `state.auth.userPhotos` / `lookHistory` / списки образов на данные с API (см. `API_REQUIREMENTS.md`).
- **Обработчики в App.tsx:** вместо `generateTryOnImage` / `generateMagicVideo` — вызовы `POST /api/tryon`, `GET /api/tryon/:id`, `POST /api/tryon/:id/video` и отображение статусов.
- **Типы:** расширить `LookHistoryItem` и т.п. полями `id`, `imageUrl`, `videoUrl` (уже есть), при необходимости — `sessionId`, `status`.
- **Константы:** убрать секреты; лимит 10 фото и TTL — могут приходить с backend или остаться в константах до первого запроса.
- **Dropzone:** после выбора файла — отправка на `POST /api/media/upload`, в callback передавать `url` (и при необходимости `id`), чтобы галереи работали с URL, а не с base64.

## 5. Компоненты, не входящие в основной поток (осторожно)

- `Gallery.tsx`, `History.tsx` — в текущем App не используются; не являются частью утверждённого потока. Не удалять без согласования; при необходимости использовать — только как reference.
- `ContentStudio.tsx`, `MagicPanel.tsx`, `ShopPanel.tsx`, `VoiceAssistant.tsx`, `PaymentModal.tsx` — не в основном потоке. Изменения только по явному требованию.

Итог: экраны «Мои фото», «Образы», «История примерок» и общая механика grid → expanded заданы перечисленными файлами; при интеграции меняются только источник данных и вызовы API, не структура экранов и не визуал.
