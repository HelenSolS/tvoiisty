# Спецификация автотестов

Один тест — одна проверка. Прошёл = функциональность в порядке, упал = нужно поправить. Тесты реализуются по этому документу позже.

---

## 1. Health (уже есть)

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| H1 | GET /health без зависимостей от DB/Redis | 200, body `{ "status": "ok" }` |

---

## 2. Миграции (Issue 2)

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| M1 | Все миграции из `migrations/*.sql` применяются подряд | Без ошибок |
| M2 | После применения существуют таблицы: users, media_assets, looks, look_likes, tryon_sessions | Запрос к information_schema или \dt |
| M3 | У media_assets есть ожидаемые колонки (id, type, original_url, preview_url, storage_key, owner_user_id, created_at, …) | Запрос к information_schema |
| M4 | Есть основные индексы и FK (например media_assets.owner_user_id → users, tryon_sessions.person_asset_id → media_assets) | Запрос к pg_constraint / pg_index |

*Запуск:* тестовая БД (docker или CI), `npm run migrate`, затем проверки M2–M4.

---

## 3. Storage (Issue 3)

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| S1 | uploadImage() возвращает объект с полями url и storageKey | Тип/формат ответа; storageKey не пустой |
| S2 | mirrorFromUrl(validUrl) сохраняет файл и возвращает наш url | 200, url с нашим доменом/storage |
| S3 | deleteAsset(path) удаляет объект; повторный get по url или list — объекта нет | После delete объект недоступен |

*Запуск:* тестовый bucket Supabase или мок; не вызывать из unit-тестов продакшен storage без изоляции.

---

## 4. Загрузка фото (Issue 4)

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| P1 | POST /api/media/upload с валидным image (multipart file) | 201, body содержит id, url, previewUrl |
| P2 | После успешного upload в media_assets есть запись с этим id, original_url / preview_url / storage_key заполнены | SELECT по id |
| P3 | Создаётся превью (отдельный файл или поле preview_url) | previewUrl в ответе и/или в media_assets |
| P4 | Запрос без файла или с не-изображением | 400 (или 422), без создания записи в media_assets |

*Запуск:* поднятые app + DB + storage (или интеграционный слой).

## 5. Мои фото — список и удаление (Issue 5)

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| F1 | GET /api/my/photos с X-User-Id возвращает только записи с owner_user_id = этот user | В ответе только фото этого пользователя |
| F2 | GET /api/my/photos без X-User-Id | 400 |
| F3 | DELETE /api/my/photos/:id с X-User-Id, где id — своё фото | 204, запись и файл удалены |
| F4 | DELETE /api/my/photos/:id с X-User-Id, где id — чужое фото | 404 (или 403), запись не удалена |

*Запуск:* app + DB, предзаполнение тестовыми users и media_assets.

## 6. Пользовательские сценарии (по CURSOR_TESTER_RULES)

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| U1 | Загрузка фото → появляется в списке GET /api/my/photos | Тот же id в ответе upload и в списке |
| U2 | Удаление фото по id → его нет в GET /api/my/photos | В списке нет этого id |
| U3 | Лимит 10 фото: после загрузки 11-го у пользователя ровно 10 записей, самое старое удалено | Count = 10; самый старый по created_at отсутствует |

## 7. Try-on и очереди (после реализации)

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| T1 | POST /api/tryon создаёт tryon_session, возвращает sessionId | 201, запись в tryon_sessions со status pending/queued |
| T2 | GET /api/tryon/:id возвращает статус и при completed — imageUrl | Статусы pending | processing | completed | failed; при completed есть imageUrl |
| T3 | Результат примерки есть в storage и привязан к сессии (result_image_asset_id) | Запись в media_assets, tryon_sessions.result_image_asset_id заполнен |
| T4 | При падении Fal срабатывает fallback (KIE); результат всё равно сохраняется | Один из провайдеров вернул результат, он в storage |

## 10. Ошибки и границы

| ID | Что проверяем | Ожидание |
|----|----------------|----------|
| E1 | Пользователь не получает сырые технические ошибки провайдера | Сообщение нейтральное (например «Ошибка генерации»), без stack/ключей |
| E2 | В логах не утекают base64, полные изображения, signed URL с секретами | Логируем только метаданные (id, status, duration) |

## Как использовать

- **Реализация:** по одному блоку (например 2 → 3 → 4 → 5), тесты писать в `backend/test/`, запуск через `npm test` и CI.
- **Падение теста:** либо чинить функциональность, либо обновить спецификацию, если контракт изменился по решению.
- **Новый функционал:** добавить в этот документ строки с ID и ожиданием, затем реализовать тест.

Можно не дублировать проверки: например, если S1 проверяет url+storageKey, в P2 достаточно проверить наличие записи в media_assets и полей, без повторной проверки формата storage.
